# Specifications 项目目录结构迁移记录

## 背景

PLM 系统为 Specifications（规格书）项目增加了产品子分类，要求在每个产品目录下建立分类文件夹：

| 中文分类 | 英文分类 |
|---------|---------|
| 通用 | General |
| 4G版 | 4G Version |
| 5G版 | 5G Version |
| 1G版 | 1G Version |
| Road | Road |
| Rail | Rail |

**要求：**
- 所有现有文件暂时先放到"通用"（中文）/ "General"（英文）分类下
- 后续由产品经理手动调整各文件到对应分类
- 需要同步更新校验脚本，确保目录结构合法

## 变更前目录结构

```
zh/
  EAP600/
    xxx.pdf
    images/
      xxx.jpg
en/
  EAP600/
    xxx.pdf
    images/
      xxx.jpg
```

## 变更后目录结构

```
zh/
  EAP600/
    通用/
      xxx.pdf
      images/
        xxx.jpg
    4G版/
    5G版/
    1G版/
    Road/
    Rail/
en/
  EAP600/
    General/
      xxx.pdf
      images/
        xxx.jpg
    4G Version/
    5G Version/
    1G Version/
    Road/
    Rail/
```

## 执行步骤

### 1. 文件迁移

使用 Python 脚本完成以下操作：

a) **创建分类目录**：在每个产品目录下创建 6 个分类子目录（中文/英文根据语言区分）

b) **移动文档文件**：将所有 `.pdf` 和 `.md` 文件从产品根目录移到"通用"/"General"目录下

c) **移动图片目录**：将 `images/`、`img/`、`imgs/` 目录移到"通用"/"General"目录下

d) **移动其他子目录**：产品目录下还有一些非标准命名的图片目录（如"四图"、"三图"、"slides"、"VG710_4G"等），一并移入"通用"/"General"目录下

e) **处理遗漏文件**：
   - `zh/ODU2002/ODU2002规格书_V1.0pdf` 缺少 `.` 扩展名分隔符，重命名为 `ODU2002规格书_V1.0.pdf`
   - 删除临时文件：`zh/VG814/parsed.json`、`zh/VG814/~$VG814...pptx`

### 2. 脚本更新

#### 新建 `scripts/本地校验/local_validate.py`

参考 template1 项目的 `local_validate.py`，新增对 sp 项目的支持：

- **目录结构检查** (`check_sp_structure`)：
  - `zh/` 和 `en/` 下只能有产品目录
  - 每个产品目录下只能有合法的分类子目录（6个中文/英文分类名）
  - 产品目录下不允许直接放文件（`.gitkeep` 除外）
  - 分类目录下只允许 `.pdf`、`.md` 文件和子目录

- **路径解析** (`resolve_sp`)：
  - 去掉语言前缀后，保留完整路径（包含分类目录名）
  - 例如：`zh/EAP600/通用/xxx.pdf` -> `EAP600/通用/xxx.pdf`
  - 分类名保持为 `Datasheets`

#### `scripts/validate_docs_path.py`

无需修改。`resolve_sp()` 函数使用 `_drop_lang()` 去掉语言前缀后直接 `join`，天然支持多级目录结构。

#### `.github/workflows/validate-and-sync.yml`

无需修改。Sync 脚本使用 `cut -d'/' -f3-` 提取文件在产品和语言之后的剩余路径，天然支持分类目录层级。

### 3. 本地验证

运行 `local_validate.py --project sp`：

- 中文文档：92 个路径校验通过
- 英文文档：138 个路径校验通过
- 目录结构检查：通过（放宽了子目录名限制，允许"四图"等非标准图片目录存在）

## 临时脚本清单

以下脚本仅用于本次迁移，迁移完成后可删除：

1. `scripts/migrate_to_categories.py` - 创建分类目录并移动文档文件
2. `scripts/migrate_images.py` - 移动 images/img/imgs 目录
3. `scripts/migrate_remaining_dirs.py` - 移动其他非标准命名的子目录
4. `scripts/migrate_remaining_files.py` - 移动产品目录下遗留的零散文件

## 注意事项

1. **路径编码**：Windows 环境下中文字符显示为乱码，但实际文件路径正确
2. **API 兼容性**：PLM API 返回 `{'result': [], 'status': 200}` 表示校验成功
3. **空分类目录**：目前除"通用"/"General"外，其他分类目录均为空，待后续手动填充
4. **Sync 行为**：推送到 documents 仓库时，路径会保持分类层级 `docs/en/EAP600/Datasheets/General/xxx.pdf`

# Manual 项目目录结构迁移记录

## 背景

PLM 系统为 Manual（用户手册）项目也增加了子分类，但和 Specifications 不同：

- **develop 分类**（开发文档）的子分类是开发相关的 SDK/API：
  - Python SDK、Docker SDK、Greengrass SDK、Device Supervisor Agent、FlexAPI

- **device-dsa / platform 分类**（用户手册）的子分类是：
  - 用户手册 / User manual
  - 安装指南 / Installation Guide
  - 快速用户手册 / Quick user manual

      xxx.md
      img/
  device-dsa/
    CPE02/
      xxx.md
      img/

    EC/
      xxx.md
      img/
  device-dsa/
    CPE02/
      xxx.md
      images/

## 执行步骤

### 1. 文件迁移

a) **develop 分类**：所有现有文件从根目录移到默认分类 `Python SDK` 下（后续由产品经理手动调整到正确分类）

b) **device-dsa / platform 分类**：所有现有文件移到 `用户手册` / `User manual` 下

c) **图片目录**：`img/` / `images/` 跟着文档一起移到对应分类下

### 2. 创建空分类目录

为每个产品目录创建所有允许的分类子目录（含 `.gitkeep`），即使当前为空，方便用户后续手动分类。

### 3. 脚本更新

#### `scripts/check_paths.py`

- **目录结构检查**：
  - develop 下产品目录只能有 `Python SDK` 等5个开发分类
  - device-dsa / platform 下产品目录只能有 `User manual` 等3个手册分类
  - 产品目录下不允许直接放文件（文件必须放在分类目录里）

- **路径解析** (`resolve_paths`)：
  - path 保留子分类层级：`EC300/Python SDK/xxx.md`
  - develop 分类：`categoryName = "Developer Documentation"`
  - device-dsa / platform：`categoryName = "Manuals"`（中英文统一）

#### `scripts/validate_docs_path.py`

- 更新 `resolve_manual`：path 保留子分类层级
- 更新结构检查函数：支持新的多级目录结构

### 4. 环境选择功能

为 `check_paths.py` 和 sp 的 `local_validate.py` 添加测试/正式环境切换：

| 环境 | 国内 URL | 国外 URL |
|-----|---------|---------|
| 正式 | poweris.inhand.online | poweris.inhandnetworks.com |
| 测试 | cloud-dev.poweris.inhand.online | cloud-test.poweris.inhand.online |

用法：
```bash
python scripts/check_paths.py --api --env test
python scripts/本地校验/local_validate.py --project sp --env test
```

### 5. 本地验证结果

**Manual 项目（测试环境）：**
